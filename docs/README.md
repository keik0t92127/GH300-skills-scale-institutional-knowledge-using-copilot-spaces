# OctoAcme プロジェクト管理 ドキュメント

## このリポジトリ/Space の目的

このリポジトリは、OctoAcme のプロジェクト管理に関わるすべてのプロセス文書を一箇所に集約し、暗黙知を形式知化するための単一の参照点（Single Source of Truth）です。各ドキュメントはバージョン管理され、検索可能な状態で維持されます。新規メンバーのオンボーディングを加速し、特定の個人への情報集中（属人化）を低減するとともに、振り返りや継続的改善のフィードバックループを支える基盤となります。

---

## OctoAcme プロジェクト管理プロセス 概要

OctoAcme のプロジェクトは **イニシエーション → 計画 → 実行・追跡 → リリース → 振り返り・継続的改善** というライフサイクルで進行します。イニシエーションフェーズでは問題定義・成功指標・ステークホルダーの特定と承認を行い、プロジェクト One-pager を作成して計画フェーズへの移行可否を判断します。計画フェーズでは優先済みバックログ・完了の定義（DoD）・マイルストーン・リスク登録を整備し、リリース計画を確定します。実行フェーズでは日次スタンドアップや週次デリバリーシンクを通じて進捗と障害を可視化し、スプリントごとのデモ・レビューで成果を確認しながら反復的にデリバリーを続けます。リリース後は振り返りで学びを抽出し、次のサイクルの改善につなげます。

プロジェクトには明確な役割分担があります。**プロジェクトマネージャー（PM）** はスケジュール・リスク・コミュニケーションを統括し、**プロダクトマネージャー（PdM）** はプロダクトビジョンとバックログ優先度を担います。**開発者（Developers）** は機能実装・テスト作成・コードレビューを行い、**QA/テスト** は受け入れ基準の充足と品質を検証します。**ステークホルダー** はインプットと承認を提供します。各役割が連携することで、顧客価値の継続的な提供と技術的負債の最小化を実現します。

コミュニケーション戦略として、PM と PdM の週次同期・デリバリーチームの週 2 回スタンドアップ・ステークホルダー向け月次更新を基本リズムとします。プロジェクトの状態はプロジェクトボードや README などの単一の情報源で一元管理し、エスカレーションはチームレベル → PM → プロダクトリード → スポンサーの順に上申します。インシデント発生時は速やかにトリアージ概要と対応アクションを共有し、事後にはブレームレスな振り返りを実施します。

品質保証は開発サイクル全体に組み込まれています。PR は 400 行以内を目安に小さく作成し、Issue リンクと受け入れ基準を明記します。CI では自動テスト・リンティング・セキュリティスキャンを必須とし、レビュー承認後にマージします。テスト戦略はユニットテスト・インテグレーションテスト・クリティカルフローのスモークテストの三層で構成されます。リリース前にはすべての受け入れ基準の充足・CI パス・リリースノートの準備・ロールバック計画の文書化を確認し、リリース後は後続検証とステークホルダーへのアナウンスを行います。
